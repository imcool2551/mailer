 S3
	내구성 좋은 스토리지, 공유 파일서버, 파일 버전관리
	버킷(프로젝트) > 폴더 > 오브젝트(파일)
	버킷이름은 고유하다
	폴더만들기. 이름을 바꾸려면 새로운 폴더를 만들고 기존폴더를 삭제해야함.
	권한을 퍼블릭으로 설정하면 파일에 url로 파일접근 가능
	파일의 특성에따라 다른 스토리지클래스 선택 (일단은 무난한 스탠다드를 쓰는걸로..)
	CDN(Content Delivery Network), Version Control 등등의 응용가능
	
 CloudFront
	캐싱서버 + CDN 
	CloudFront가 웹클라이언트와 기존서버 사이에 위치. 기존서버는 Origin이고 CloudFront는 Distribution.
	캐시가 유효한동안은 브라우저는 캐시서버에 요청을 하지않고 캐시서버는 오리진서버에 요청을 하지 않는다
	Origin이 웹서버라면 DomainName에 오리진의 도메인만(!!) 설정, 포트번호 유의
	CloudFront의 도메인이 곧 캐싱서버의 도메인
	캐시의 TTL은 Origin에서 헤더를통해 (Cache-Control: max-age=60) 설정하고 동시에 CF의 Behavior 탭에서 설정할 수 있음
	Invalidations 탭에서 캐시삭제 (유료)

 Route53
	DNS 등록대행자 + A레코드 관리하는 네임서버 임대
	1단계. 등록대행자는 등록소에서 도메인을 사준다
	2단계. 등록대행자는 도메인의 A레코드를 관리한다. 등록소는 등록대행자의 네임서버인 NS레코드를 관리한다.
	3단계. 우리컴퓨터는 루트 DNS Server를 이미 알고있다.
	도메인구매대행, 도메인이전 둘 다 가능 (도메인이전만 살펴보자!)
	가장 먼저, 호스팅영역 생성을 통해 Route53에게 도메인을 관리하는 네임서버를 임대 받는다
	Route53 에서 도메인을 구매하지 않은 경우 우리 도메인을 관리하는 네임서버를 변경해야한다
	도메인의 네임서버 변경은 도메인을 구입한 사이트에서 해준다
	dig +trace [url] 로 네임서버를 확인할 수 있다
	dns - ip 매핑은 이후에 Route53에서 레코드세트생성을 통해 해준다. (TTL은 캐시)
	Route53은 Alias를 통헤 AWS의 다른 서비스 (S3, ELB, CF) 와 쉽게 연동가능
	LB에 응용가능...
